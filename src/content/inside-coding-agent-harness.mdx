> "It's just calling the API in a loop."

Every developer who's poked at Claude Code or Codex CLI for the first time eventually says this. And they're not wrong — but they're describing maybe 5% of the system. The other 95% is what makes the difference between a toy that occasionally works and an agent that can actually change production code without destroying your repo.

This post tears open that 95%. Not a tutorial on how to use Claude Code — an autopsy of what actually runs when you type a prompt into any serious coding agent. We'll look at the harness, the tool execution pipeline, context management, memory, and the permission layer. By the end, you'll understand what problems each layer solves and why they exist.

---

## The Misconception: It's Not a Chatbot

The mental model most people carry is this: user sends message → LLM responds → done. That works for a conversational assistant. Coding agents are different. They operate in a loop, take actions with real side effects, and have to manage state across an unbounded number of turns without losing coherence or destroying the user's codebase.

The moment you add side effects — file writes, shell commands, web fetches — you've left chatbot territory. You need a runtime that can:

- **Execute** tool calls safely and in the right order
- **Track** what happened across turns
- **Decide** what context to preserve and what to discard
- **Gate** potentially destructive operations on user approval
- **Recover** gracefully when the model hallucinates a tool call or a command fails

That runtime is the harness. Everything else is built on top of it.

---

## The Harness: The Runtime, Not the Model

The harness is the orchestration layer that wraps the LLM. The model only sees text — it cannot execute code, touch files, or call APIs on its own. The harness bridges the model's text output into real system operations and feeds the results back.

![Coding agent harness architecture — data flow from user input through intent router, LLM loop, tool executor, permission policy, OS sandbox, compaction, and SQLite persistence](./diagram-harness.png)

Think of it as a thin but deeply opinionated process manager. A minimal harness contains:

```
AgentRuntime {
  messages: Message[]       // the only state
  llmClient: LLMClient      // provider-agnostic
  toolExecutor: ToolExecutor
  permissionPolicy: Policy
  systemPrompt: string
  usageTracker: UsageTracker
}
```

There is no explicit state machine. No workflow graph. The message array *is* the state. The entire agent's working memory lives in that array.

Claude Code's harness, for instance, separates reasoning (the model) from mediation (everything else) across six distinct layers:

1. **Input Layer** — session management, permission modes, project trust gates
2. **Memory Layer** — CLAUDE.md, project notes, skill files
3. **Execution Layer** — tool definitions, parallel vs. sequential scheduling, cost tracking
4. **Integration Layer** — MCP servers, external capabilities
5. **Multi-Agent Layer** — subagent spawning, worktree isolation
6. **Observability Layer** — lifecycle hooks (PreToolUse, PostToolUse, Stop, PreCompact)

Each layer has a specific job and doesn't bleed into the others. That separation is what allows the system to be composable — you can swap out the permission policy without touching the tool executor.

## The Agent Loop

The loop is simple by design:

```
while not done:
  response = llm.complete(messages, tools)
  messages.append(response)
  
  if response.is_text_only:
    break
  
  for tool_call in response.tool_calls:
    result = execute(tool_call)
    messages.append(tool_result(result))
  
  if budget_exceeded or turn_limit_reached:
    break
```

That's it. Every coding agent — Claude Code, Codex CLI, Cursor — runs some variant of this. The complexity is entirely in what `execute(tool_call)` does and how `messages` is managed.

What makes this different from a simple API call:

- **Tool calls can fail.** The harness must decide: retry, report error to model, or abort?
- **Tool results can be enormous.** A `cat` on a 50k LOC file would blow the context window. The harness must truncate or summarize.
- **Some tools must run sequentially.** File writes must not race. The harness schedules read-only tools in parallel and serializes mutating ones.
- **The loop can run indefinitely.** Budget and turn limits are not optional guardrails — they're structural constraints on a system that has no natural stopping condition.

## Tool Calls: The Real Complexity

A tool in an agent harness is not just a function. It's a contract between the model and the runtime, with three components:

### 1. The Schema

The harness ships tool definitions to the LLM as JSON Schema. The model's tool calls must conform to these schemas — any deviation is a structured output failure that the harness must handle.

```json
{
  "name": "bash",
  "description": "Execute a shell command in the project directory",
  "input_schema": {
    "type": "object",
    "properties": {
      "command": { "type": "string" },
      "timeout": { "type": "number" }
    },
    "required": ["command"]
  }
}
```

The model never sees actual function signatures. It sees text descriptions of what tools do. The quality of these descriptions directly affects how reliably the model invokes tools correctly.

### 2. The Execution Pipeline

When the harness receives a tool call, it runs through a pipeline before touching the system:

```
tool_call
  → parse & validate input
  → check permission policy
  → request user approval (if required)
  → execute in sandbox
  → capture stdout/stderr/exit code
  → format result for model context
  → append to message array
```

The core tool taxonomy breaks into four classes, and the harness treats each fundamentally differently:

| Class | Examples | Scheduling | Approval |
|-------|---------|-----------|---------|
| **Read** | ReadFile, Grep, Glob, LS | Parallel | Auto — truncate if oversized |
| **Write** | Edit, Write, MultiEdit | Sequential | Always explicit, show diff first |
| **Execute** | Bash | Sequential | Static analysis first, then approval |
| **Web** | Search, OpenPage, FetchURL | Parallel | URL allowlist or per-request |

Read tools run in parallel safely. Write tools must serialize — concurrent edits to the same file corrupt state. Bash is the most dangerous: one command can delete the repo, exfiltrate credentials, or fork-bomb the machine. Web tools are parallel but introduce a different risk class: outbound network calls.

**Web search is the tool that patches a structural limitation of the model itself.** LLM training data has a hard cutoff — everything published after it is invisible. For a coding agent this matters concretely: the npm package the model confidently recommends may be deprecated, the API signature may have changed, the CVE it doesn't know about may be in the code it just wrote. Without web search, the agent reasons with frozen training data. With it, results get injected into the message array as tool results and the model's next response is grounded in live reality.

The harness decides *when* to search in one of three ways: automatic triggers (any question about a library version), model-driven (model calls the search tool when it decides it needs context), or user-explicit (a `/web` command). Each has a failure mode. Automatic triggers over-fire. Model-driven fails when the model doesn't know what it doesn't know — if training data confidently contained wrong information, the model won't think to verify. User-explicit requires the user to notice the gap. Archer uses model-driven and exposes Tavily, Exa, and its own Archer Scout as pluggable providers — the same provider-abstraction pattern as the LLM layer.

When I built Archer — my own attempt at replicating this harness from scratch — I implemented three approval modes to see how far you can get with a simple policy instead of per-tool rules:
- `read-only` — denies all writes automatically
- `workspace-write` — writes require user approval, reads are automatic
- `danger-full-access` — auto-approves everything (for trusted automation)

### 3. The Approval Layer

Approval is where the harness makes value judgments about risk. A naive implementation asks the user about everything. A well-designed one uses a policy engine.

In Archer, this lives in a small `@archer/sandbox` policy module:

```typescript
decidePathAccess(path: string, operation: "read" | "write"): "allow" | "ask" | "deny"
decideCommand(command: string): "allow" | "ask" | "deny"
```

`decideCommand` runs a shell command through a static analyzer — it tokenizes the command and checks for known-dangerous patterns (pipes to `rm`, `chmod 777`, `curl | bash`, etc.) without executing it. This is not foolproof, but it catches the obvious cases before they reach the user prompt queue.

The policy is separate from the tool executor by design. You can swap in a stricter or more permissive policy without changing a single line of tool code.

## The Sandbox: Two Layers, Not One

![Coding agent sandbox — permission policy at the app layer and OS sandbox at the kernel layer operating independently](./diagram-sandbox.png)

The permission policy and the OS sandbox are two different systems solving two different problems. Most people treat them as one thing. They're not.

The **permission policy** lives at the application layer — it answers *"should we run this?"* before execution begins. The **OS sandbox** lives at the kernel layer — it answers *"even if we run it, what can it actually touch?"* during execution. A tool call can pass the permission check and still be constrained by the sandbox. Independence is the point: if the policy layer makes a mistake, the sandbox catches it. If the sandbox has a misconfiguration, the policy is a first line of defense. Defense in depth only works when the layers fail independently.

Claude Code implements OS-level sandboxing via macOS `sandbox-exec` and Linux namespaces — kernel-enforced boundaries that restrict which filesystem paths the process can read/write and which network endpoints it can reach. The application-level permission model doesn't replace this; it sits above it.

**Layered defenses can degrade in non-obvious ways.** In production, Claude Code hit a real case: commands with more than 50 subcommands fell back to a single generic approval prompt instead of running per-subcommand policy checks, because granular parsing caused UI freezes. The security property was silently weakened under real load — two layers sharing a failure mode through the UI thread.

The consequences of getting the sandbox wrong are concrete. A now-patched SOCKS5 null-byte injection in Claude Code's network sandbox affected every release from v2.0.24 through v2.1.89 — roughly 130 versions over 5.5 months. It allowed arbitrary outbound network calls from inside the sandbox: credentials, source code, anything accessible to the process. Anthropic patched it silently in v2.1.90 with no mention in release notes.

In Archer, I only built the policy layer — no OS-level sandbox. For a local, trusted environment with explicit approval modes it's fine. But the gap is real: a sufficiently crafted bash command could still escape policy checks in edge cases. For anything automated or multi-user, the OS sandbox layer is not optional.

## Memory: Three Completely Different Problems

"Memory" in an agent harness means three different things, and conflating them is where most naive implementations go wrong.

![Coding agent memory layers — from volatile message array through compaction artifact to persistent SQLite and permanent project files](./diagram-memory.png)

### 1. In-Session Context (The Message Array)

The most immediate form of memory is the conversation history living in the message array. This is what the LLM actually sees. It grows with every turn — user messages, assistant responses, tool calls, tool results. Tool results are the real problem: a single `grep` or `find` can add thousands of tokens.

The context window is finite. In a long coding session, tool observations can consume 70-80% of available tokens. When the array grows too large, the model starts "forgetting" earlier context — or the API call simply fails with a 400.

### 2. Cross-Session Persistence (Storage Layer)

The second problem: when a session ends, everything in the message array disappears. For a coding agent that works across days or weeks on a project, this is catastrophic.

Serious agents solve this with a persistent storage layer. In Archer, I went with SQLite + Drizzle ORM — four tables covers the essentials:

| Table | Purpose |
|-------|---------|
| `sessions` | Session metadata, provider, model, CWD |
| `messages` | Full transcript (user/assistant/tool) |
| `model_messages` | LLM conversation history for reconstruction |
| `turn_results` | Per-turn metrics, intent, status, summaries |

On session resume, the harness reconstructs the message array from storage. The model sees a coherent conversation history; the user gets continuity across restarts.

### 3. Project Knowledge (Durable Files)

The third problem is the hardest: how does the agent know project-specific conventions, architecture decisions, and constraints that weren't in any conversation?

Claude Code uses `CLAUDE.md` — a markdown file at the project root prepended to every system prompt. Other agents (Codex, Cursor, GitHub Copilot) use `AGENTS.md`, the open standard OpenAI originated for Codex CLI in 2025 and donated to the Linux Foundation. Different filenames, identical concept: a human-maintained file that survives compaction because it lives on disk, not in the message array.

One non-obvious research finding on AGENTS.md across 138 real repos: LLM-generated versions *reduce* agent task success rates and increase inference cost by 20%+. Human-written minimal ones give only +4% improvement — and only when precise. The file should contain only what the agent cannot infer from the codebase. Everything else is noise that actively degrades performance.

In Archer, `.agents/config` serves the same role. The pattern is identical across every serious agent: durable files on disk become persistent project knowledge, immune to compaction.

## Context Compaction: Controlled Amnesia

When the context window fills up, the harness has to decide what to forget. This is the compaction problem, and different agents solve it differently.

The naive solution — truncate from the front — destroys coherence. The model loses the original task, early constraints, and half the conversation.

Serious implementations use a summarization strategy:

```
compaction_trigger:
  if tokens_used / max_tokens > 0.92:
    1. drop old tool results (they're high volume, low signal)
    2. summarize the dropped conversation with a separate LLM call
    3. replace dropped messages with the summary artifact
    4. keep CLAUDE.md / AGENTS.md on disk (they're never compacted)
    5. continue
```

Claude Code fires at ~92% capacity. In Archer, `compaction-policy.ts` splits the message array into `protectTokens` (recent context, kept verbatim) and `prunableTokens` (older context, summarized into a `CompactContinuationArtifact`).

The key insight is that tool results decay in value much faster than assistant reasoning. A file listing from 20 turns ago is noise. The decision the agent made based on that listing is signal. Compaction should preserve decisions and discard raw observations.

## The Two-Phase Routing Problem

Not every user message requires the full agent loop. "What does this function return?" needs a single LLM call. "Refactor the auth module" needs 30 tool calls across 15 turns. Treating these the same is wasteful and slow.

In Archer, I built explicit intent routing to handle this:

**Phase 1: Fast-path detection**

Syntactic heuristics classify the input before touching the LLM:
- Single sentence question → `direct-answer`
- URL or "search for" → `web-context`
- File mention + question → `repo-context`
- Imperative verb (fix, add, refactor, implement) → `change`

**Phase 2: LLM classification (fallback)**

If heuristics are ambiguous, a small, fast model resolves the intent via a structured `submitTurnDecision` tool call. This costs tokens but runs once at the start of the turn, not repeatedly.

Each intent maps to a different execution path with different tool sets, different approval thresholds, and different token budgets. A `direct-answer` turn never touches the filesystem. A `change` turn gets full tool access.

This is the difference between an agent that feels responsive and one that kicks off a 30-second planning phase for every input.

## Multi-Provider LLM Abstraction

Production coding agents don't hardcode to one provider. The model you want for fast interactive coding (Gemini Flash) is not the same as what you want for complex refactoring (Claude Opus). Users have API keys for different providers. Costs vary by an order of magnitude.

The right abstraction is a provider-agnostic interface that resolves at runtime:

```typescript
resolveLanguageModel({ provider, modelId, apiKey }): LanguageModel
```

In Archer, I used the Vercel AI SDK for this — unified interface across OpenAI, Anthropic, Google, DeepSeek, and OpenRouter with one consistent `streamText` call. Provider resolution reads from environment variables:

```
ARCHER_PROVIDER=anthropic
ANTHROPIC_API_KEY=sk-ant-...
AGENT_MODEL=claude-3-5-sonnet-latest
```

The LLM call itself is identical regardless of provider. The harness doesn't know or care which model it's calling — it sends messages and tool definitions, receives a response with optional tool calls, and continues the loop.

One non-obvious optimization here: prompt caching. Stable system prompts and tool definition arrays that don't change between turns can be cached at ~90% cost reduction on repeated reads. This is structural — you don't cache specific responses, you cache the *prefix* that's identical across every call in a session.

Model choice matters here beyond benchmarks. DeepSeek R1/V4 is a good example: it consistently performs well in harness contexts despite a smaller training corpus than frontier closed-source models. The reason is that it was RL-trained against actual sandboxed agentic environments at scale — tool call schema compliance gets optimized as an outcome, not pattern-matched from examples. For a harness that depends entirely on reliable structured output, that training signal matters more than raw knowledge breadth.

## The TUI Layer: Making It Usable

A terminal UI for an agent is not a cosmetic decision. It's load-bearing infrastructure.

The agent loop produces a stream of events: tool invocations, token streams, approval requests, compaction events, errors. Without structured rendering, this is a wall of JSON. The TUI translates that event stream into something a developer can actually monitor and respond to.

The critical moments the TUI must handle:

1. **Approval prompts** — the user must see exactly what the agent is about to do and make a real-time decision. A bad UX here means users rubber-stamp everything, defeating the permission system.
2. **Streaming output** — LLM responses stream token by token. Buffering to completion before display creates a worse experience than streaming, but raw streaming needs layout management.
3. **Tool result display** — showing a full 500-line file diff vs. a summary vs. nothing is a UX decision with real consequences for how much the user understands what's happening.
4. **Cost tracking** — token counts and estimated USD cost should be visible after every turn. The user needs to know what they're spending.

In Archer, the TUI is built on OpenTUI — a React-inspired terminal rendering framework. Events from the harness map to UI components: tool invocation blocks, approval dialogs, streaming text panels, cost footers. The TUI never blocks the agent loop; it renders asynchronously while execution continues.

## A Single Turn, Unwrapped

Abstract architecture is easier to understand with a concrete example. Here's exactly what happens when you type `"fix the auth bug in auth.ts"` into a coding agent:

**1. Intent routing** — the harness sees an imperative verb (`fix`) and a specific file. Fast-path heuristic: classify as `change`. No LLM call yet.

**2. Context assembly** — harness loads AGENTS.md / CLAUDE.md, recent session messages from SQLite, and the tool definitions array. Builds the system prompt. Checks if tokens are already near threshold — if so, compacts before the first LLM call.

**3. First LLM call** — model receives system prompt + message history + tool definitions. Responds with two tool calls: `ReadFile("auth.ts")` and `Grep("authenticate", ".")`.

**4. Parallel execution** — both are read tools. Harness runs them in parallel. Permission policy auto-approves (reads are never blocked). Results come back: file contents + grep matches.

**5. Results appended** — tool results added to message array. Second LLM call made with updated context.

**6. Model plans the fix** — responds with `Edit("auth.ts", patch)`. This is a write tool.

**7. Policy check** — harness hits the permission policy: file write → `ask`. Policy calls `decidePathAccess("auth.ts", "write")` → `ask`. TUI renders a diff of the proposed change.

**8. User approves** — harness applies the patch. Tool result appended. Third LLM call.

**9. Verification** — model calls `Bash("npm test")`. Shell execution → static analysis runs first. Checks for dangerous patterns: none found. Still `ask` because it's a shell command. TUI shows the command. User approves.

**10. Sandbox execution** — command runs inside OS sandbox. Filesystem and network restrictions apply even though it was approved at the policy layer.

**11. Tests pass** — result appended. Fourth LLM call. Model produces a text-only response (no more tool calls) — that's the exit condition for the loop.

**12. Persist** — turn result written to SQLite: intent (`change`), status (`completed`), token count, estimated USD cost. TUI renders the final summary.

Twelve distinct steps. Four LLM calls. Two approval prompts. One loop iteration per LLM call. Every step lives in a different layer of the harness.

## Building Your Own: What the Stack Actually Looks Like

If you wanted to build a coding agent with this architecture today, the stack choices that matter:

**LLM abstraction:** Vercel AI SDK. Provider-agnostic, handles streaming, structured output, and tool call formatting across all major providers. Saves weeks of integration work.

**Agent core:** `@openharness/core` or roll your own loop around `streamText` with tool execution callbacks. The loop itself is 100 lines; the infrastructure around it is 10,000.

**Storage:** SQLite for local-first persistence. Drizzle ORM for type-safe schema management. Session and message tables are table stakes; turn metrics and compaction artifacts require design.

**Terminal UI:** OpenTUI (React bindings for the terminal) or Ink. Both let you write declarative component trees that render to ANSI. State management is identical to a web app — events flow in, components render.

**Sandbox/policy:** Write your own. The logic is simple but the judgments are project-specific. Static shell command analysis + path escaping detection + configurable approval modes covers 95% of cases.

**Tool set:** Start with bash, file read/write, and git. Web fetch and MCP integration can be added later. The tool schema design matters more than the tool count — a well-described bash tool beats six poorly described specialized tools.

The infrastructure is not glamorous. Compaction policy, approval queues, cost tracking, session resume — none of this shows up in demos. But it's what separates a weekend project from something you can actually run on a real codebase.

## The Real Architecture Insight

Here's the thing that took a while to internalize: the model is the smallest part of the system.

The clearest proof: take the exact same model and run it in two different harnesses. Claude Sonnet in Claude Code completes a benchmark task in **33K tokens**. The same Claude Sonnet in Cursor uses **188K tokens** for the identical task — 5.5x more. Same model. Different context management, different tool schema design, different compaction strategy. The harness accounts for the entire difference.

On SWE-bench Verified, Claude Code hits 72.5%. Cursor with the same underlying Claude model sits at 55-62%. That gap is not the model — it's twelve steps of harness engineering applied differently.

And it cuts the other way too. Claude Opus 4.7 scores **91.1% in Cursor's harness** versus 87.2% in Claude Code's own harness. Cursor's context orchestration sometimes extracts more from a model than the model maker's own tooling. The harness gap is now larger than the model tier gap.

Swapping GPT-4o for Claude Sonnet in a well-built harness takes ten lines. Replacing the permission policy, improving compaction, or redesigning tool schemas takes weeks. The quality of a coding agent is almost entirely a function of harness engineering — context management, approval UX, tool design, routing decisions.

The LLM is a black box with a text interface. The harness is the engineering.

## Archer: My Attempt at Replicating This

Everything in this post came from building [Archer](https://github.com/adarshaacharya/archer) — my own attempt to understand how a coding harness actually works by writing one from scratch. It's not competing with Claude Code or Codex CLI. It's a learning project that forced me to confront every problem described above: compaction thresholds, approval queues, provider abstraction, intent routing, session persistence.

It's BYOK, multi-provider, runs in the terminal, and has all the pieces — but the honest framing is that it's a study in harness engineering, not a frontier product. Building it taught me that compaction policy is harder than it looks, that the approval UX determines whether users actually trust the agent, and that context management is where every naive implementation eventually breaks.

If you want to understand what makes these systems tick, reading a working implementation is faster than reading architecture docs.
